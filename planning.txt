=========================================================
PHASE 1 — PROJECT INITIALIZATION
================================

QUESTION:
Why did we initialize Node.js project first?

ANSWER:

Because backend needs a runtime environment.

Node.js provides:

* JavaScript runtime
* package ecosystem
* server execution environment
* async architecture

Without initialization:

* no package management
* no dependency tracking
* no scripts
* no project structure

So we created:

package.json

This became the identity card of backend project.

=========================================================
WHY TYPESCRIPT?
===============

QUESTION:
Why TypeScript instead of plain JavaScript?

ANSWER:

Because large backend systems become hard to manage.

Problems in JavaScript:

* no type safety
* runtime bugs
* harder refactoring
* difficult scaling
* difficult team collaboration

TypeScript solves:

* type checking
* autocomplete
* safer refactoring
* predictable architecture
* scalable codebase

Example thinking:

If a function expects:

email: string

then TypeScript immediately warns if number/object is passed.

This prevents production bugs.

=========================================================
WHY MIDDLEWARES?
================

QUESTION:
What is middleware?

ANSWER:

Middleware sits between:

request → response

It can:

* modify request
* validate request
* authenticate user
* log request
* block malicious requests

Flow:

incoming request
↓
middleware
↓
controller
↓
response

=========================================================
WHY CREATE FOLDERS?
===================

QUESTION:
Why not put everything inside one file?

ANSWER:

Because projects become massive.

If all logic stays inside app.ts:

* impossible to debug
* impossible to scale
* impossible for teams
* difficult testing
* difficult maintenance

So backend engineering separates concerns.

=========================================================
WHY CONTROLLERS?
================

Controllers handle:

* request
* response

Controller SHOULD NOT contain heavy business logic.

Controller acts like:

"traffic manager"

Example:

req.body received
↓
send data to service
↓
return response

=========================================================
WHY SERVICES?
=============

This is VERY IMPORTANT.

Services contain:

BUSINESS LOGIC.

Example:

login user

create post

Why separate?

Because:

business logic should be reusable.

Controllers should remain thin.

=========================================================
ACCESS TOKEN VS REFRESH TOKEN
=============================

Access Token:

* short-lived
* used for APIs
* expires quickly

Refresh Token:

* long-lived
* generates new access token
* stored securely

This architecture is used by:

* SaaS apps
* enterprise systems
* banking apps
* modern React applications

=========================================================
WHY COOKIE SECURITY?
====================

We used:

httpOnly cookies

Because:

JavaScript cannot access them.

This protects against XSS attacks.

We used:

sameSite: strict

to reduce CSRF risks.

We used:

secure cookies

so cookies travel only over HTTPS in production.

=========================================================
WHY HASH REFRESH TOKENS?
========================

Same reason passwords are hashed.

If database leaks:

attacker should NOT directly use refresh tokens.

So:

raw token
↓
hash token
↓
store hash only

Security engineering mindset.

=========================================================
WHY REQUEST ID MIDDLEWARE?
==========================

Every request now gets:

req_abc123

Purpose:

Tracing.

Now every log contains same request ID.

This helps:

* debugging
* monitoring
* production tracing
* distributed systems later

Huge backend engineering concept.

=========================================================
WHY MORGAN LOGGER?
==================

Morgan logs:

* incoming requests

* response time

Useful for:

* debugging
* monitoring traffic
* development visibility

=========================================================
WHY BRUTE FORCE PROTECTION?
===========================

Attackers try:

1000 passwords repeatedly.

We implemented:

* failed login tracking
* temporary lock
* retry window

This prevents automated attacks.

Production security concept.
